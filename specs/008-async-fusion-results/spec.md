# Feature Specification: Async Fusion Results via Deferred Retrieval

**Feature Branch**: `008-async-fusion-results`

**Created**: 2026-06-19

**Status**: Draft

**Input**: User description: "A deferred-result mechanism so long fusions stop timing out in clients that don't support MCP Tasks (codex, ZCode). When a fusion will exceed the client's tool-call ceiling, return immediately with a reference id and an instruction to retrieve the result later via the same tool; the result is fetched on subsequent calls. Durable (SQLite) for both parallel and sequential modes."

## Background & Motivation

OpenFusion runs as an MCP server that a client drives. MCP is fundamentally request/response: a client sends one `tools/call`, and the server must return the full result inside that single round-trip. Most MCP clients enforce a per-call timeout — codex's is **~60 seconds by default** (`active_time_timeout` in `run_service_operation`), and it is widely reported as sticky at that value. A single OpenFusion fusion can take anywhere from ~60 s (parallel, a few cloud candidates) to **21+ minutes** (sequential mode with local models — see spec 007). When the fusion's wall-clock exceeds the client ceiling, the client kills the call mid-work — **even though the work succeeds server-side and the answer is computed and thrown away.**

Feature 005 (MCP Tasks / SEP-1686) attempted to solve this by detaching the work and returning a `CreateTaskResult` immediately, with the client fetching the result later via `tasks/get` and `tasks/result`. That works **only if the client implements the Tasks client**. Source review of codex (`codex-rs/rmcp-client/src/rmcp_client.rs`, `call_tool()`) confirms codex does **not**: it hardcodes `task: None` on every `tools/call`, only handles `ServerResult::CallToolResult` (a `CreateTaskResult` would be an `UnexpectedResponse` error), and exposes no `tasks/get` / `tasks/result` methods on `RmcpClient` at all. The codex binary contains the Tasks strings only because it links the `rmcp` SDK that defines them — they are dead weight from a dependency, not behavior. ZCode behaves identically (non-Tasks client). So feature 005's headline success criterion ("stops the timeout for codex") is **unreachable as built**: non-Tasks clients still block for the full fusion and still time out.

This feature adds the missing piece: a **client-driven retrieval protocol** that works for any MCP client, including those that never send `params.task`. The mechanism is an LLM-mediated round-trip on the same `fusion` tool:

- On kickoff, `fusion` returns immediately with a **reference id** plus an imperative instruction to retrieve the result later via `fusion({ _resume_from: "<id>" })`.
- On retrieval, the agent calls `fusion({ _resume_from: "<id>" })` and gets either the synthesized answer (if done) or a "still working, call again" result (if not).
- The fusion's real work runs **detached** — the same `startDetachedFusion` runner feature 005 already builds — so no single `tools/call` stays open for the whole wall-clock.

This is explicitly the pattern SEP-1686 rejected as "expensive and inconsistent" (LLM-mediated polling). The panel consultation confirmed that rejection **only applies when a protocol alternative exists**; for non-Tasks clients it does not. "When the bridge is out, drive around." For a parallel fusion (~60–140 s), this means 1–3 bounded long-poll calls instead of one hung call. For a sequential fusion (~12–21 min), it is fire-and-forget on kickoff plus one retrieval near completion (not tight polling — see US2).

The feature is **mode-aware** because spec 007 (sequential processing) changes the constants by 10–20×. A single retrieval cadence breaks both ways: always-tight-poll token-storms sequential jobs; always-fire-and-forget adds friction to ~90 s parallel jobs. So the kickoff and retrieval message shapes, the time-to-live (TTL), and the "tell the user it's pending" wording all adapt to execution mode.

**Storage decision (locked).** Feature 005 deliberately scoped result storage as non-durable ("in-memory, process-restart loses in-flight tasks — accepted"). That acceptance breaks for this feature, because a 21-minute sequential job has a high probability of overlapping a restart, crash, or deploy — turning into an orphan ticket plus wasted provider compute. The panel's initial hybrid ("parallel in-memory, serial SQLite") optimizes machinery-per-mode. This feature chooses the simpler, more robust landing: **durable (SQLite) storage for both parallel and sequential result/state records, and ephemeral (in-memory) live progress for both modes.** This is one retrieval path, one TTL/eviction story, and one identity (the SQLite activity id is the reference id — no three-way map). The only thing kept ephemeral is live "candidate 3 of 5" progress, which on restart describes nothing (the job died with the process).

This feature also keeps feature 005's Tasks path intact for clients that do support it (the SDK's auto-polling TS clients get the zero-LLM-cost host-poll path). The Tasks path and this `_resume_from` path share the same detached runner and the same durable result store; only the egress shape differs.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Parallel fusion on a non-Tasks client no longer times out (Priority: P1)

A user drives OpenFusion through a non-Tasks client (codex or ZCode) with a ~90-second parallel fusion of 3–5 cloud candidates. Today this times out: the client's ~60 s per-call deadline fires mid-fusion, the answer is computed server-side and discarded, and the user sees a timeout error. After this feature, the first `tools/call` returns **≈1s** (well under any plausible client timeout) with a result that contains a reference id and an imperative instruction ("You MUST call `fusion({ _resume_from: '<id>' })` to receive the result…"). The agent calls `_resume_from`, which **bounded-long-polls** (waiting up to a sub-ceiling duration — default ~40 s — and returning the answer if it lands in that window, or a "still working, call again" result if not). For a 90 s fusion that is typically one kickoff + one or two retrieval calls. The user gets the synthesized answer with no timeout.

**Why this priority**: This is the core, production-visible win that feature 005 promised but could not deliver for non-Tasks clients. Without it, OpenFusion remains unusable on codex/ZCode for any fusion exceeding ~60 s. Every other story is in service of making this robust across modes and over the process lifecycle.

**Independent Test**: Configure a real (or faux-provider) fusion that takes a known ~90 s, call `fusion({ prompt })` from a synthetic non-Tasks client (one that sends no `params.task`), and verify (a) the first call returns ≈1s with `{ status: "processing", reference_id, instruction }`, (b) the work is detached and continues server-side, (c) subsequent `fusion({ _resume_from })` calls each wait up to ~40 s and return either `{ status: "completed", result }` or another `processing` result, (d) the final synthesized answer is byte-identical to the synchronous path's output for the same inputs. E1 (spec 005's never-run end-to-end test) is re-scoped and finally runs here, against a real client.

**Acceptance Scenarios**:
1. **Given** a non-Tasks client and a fusion whose wall-clock exceeds the client ceiling, **When** the agent calls `fusion({ prompt })`, **Then** the `tools/call` returns within a small constant (well under any plausible client timeout) carrying `{ status: "processing", reference_id, retry_after_ms, instruction: <call _resume_from> }`, and no provider work is serialized into the call.
2. **Given** an in-progress fusion with `reference_id = R`, **When** the agent calls `fusion({ _resume_from: "R" })`, **Then** the call bounded-long-polls up to the configured wait and returns `{ status: "completed", result: <synthesized answer> }` if the fusion finishes within the wait, else `{ status: "processing", reference_id: "R", retry_after_ms, instruction: <call _resume_from again> }`.
3. **Given** the fusion has completed and its result is still within TTL, **When** the agent calls `fusion({ _resume_from: "R" })`, **Then** the call returns `{ status: "completed", result }` promptly (no waiting).
4. **Given** a Tasks-aware client (one that sends `params.task`), **When** it calls `fusion`, **Then** behavior is unchanged from feature 005 — it receives a `CreateTaskResult` immediately and retrieves via `tasks/result`; the `_resume_from` path is not used.

### User Story 2 - Sequential (long) fusion uses fire-and-forget retrieval, not tight polling (Priority: P1)

A user with sequential mode enabled (spec 007) kicks off a 12–21 minute serial fusion from a non-Tasks client. Tight polling is untenable here: ~22 retrieval calls at 40 s each for a 15-minute job is exactly the token-storm SEP-1686 warned about, and the agent will lose the thread or exhaust the context window long before completion. Instead, the kickoff returns immediately with a reference id, an honest **ETA** (derived from spec 007's serial budget formula), and a message that tells the user (via the agent) that this is a long job — including a dashboard link where progress can be watched live. The agent does **not** tight-loop; it either retrieves once near the ETA or hands the user the ticket and the dashboard URL to come back later. When the agent does retrieve, the result is either ready or the ETA is refined.

**Why this priority**: Sequential mode (spec 007) is otherwise unusable from any non-Tasks client — a 21-minute wall-clock dwarfs every client's ceiling, and tight polling would burn tokens and context. This story makes spec 007 reachable from codex/ZCode. It is co-priority with US1: together they cover both execution modes.

**Independent Test**: Enable sequential mode with a candidate count that yields a known serial budget (~15 min). Call `fusion({ prompt })` from a synthetic non-Tasks client and verify (a) the kickoff returns immediately with `{ status: "processing", reference_id, eta_ms, dashboard_url, retry_after_ms, instruction }`, (b) the instruction for sequential mode tells the user about the long duration and includes the dashboard link (the wording is user-facing, not the terse parallel shape), (c) a single `_resume_from` call well before the ETA returns `processing` with a refined ETA, (d) a `_resume_from` call after completion returns the synthesized answer. Verify the retrieval call count for the whole job stays small (single digits), not proportional to wall-clock.

**Acceptance Scenarios**:
1. **Given** sequential mode is ON, **When** the agent calls `fusion({ prompt })`, **Then** the kickoff result includes `eta_ms` (derived from spec 007's serial budget formula for the current candidate count) and a `dashboard_url` pointing at the live-status surface from spec 007.
2. **Given** a sequential fusion is in progress, **When** the agent calls `fusion({ _resume_from })`, **Then** the result tells the user (via the agent) to come back later / check the dashboard, rather than instructing a tight re-poll — the retrieval cadence for sequential mode is ETA-guided, not tight.
3. **Given** the fusion's ETA is known, **When** the agent retrieves before the ETA, **Then** the response includes a refined (remaining) ETA rather than a generic "still working".
4. **Given** parallel mode is ON, **When** the agent retrieves, **Then** the response uses the tight-poll cadence with terse transparent-pacing wording (retrieval mandate + `retry_after_ms`, no "do not inform the user" directive) appropriate to a short job — confirming the message shape and cadence are mode-dependent.

### User Story 3 - A restart during a long fusion does not orphan the ticket or the compute (Priority: P1)

A user kicks off a 21-minute sequential fusion (or a 90-second parallel fusion while a deploy lands) and OpenFusion restarts while the job is in flight. The in-memory result store feature 005 used would lose the job silently — the next `_resume_from` would return "not found," the user has no answer, and the provider compute already spent on partial candidates is wasted. After this feature, the job's state and (on completion) its result are persisted to SQLite, so a restart is recoverable: a retrieval after restart returns a defined "job was interrupted" outcome with a clear instruction (re-run or check the dashboard), and a job that finished just before the crash still has its answer retrievable until TTL. Live candidate-by-candidate progress is **not** persisted (on restart it describes nothing — the job died), which is the one deliberately ephemeral surface.

**Why this priority**: A long fusion that silently disappears on restart is a correctness failure, not a polish item. Without durability, US1 and US2 are fragile under the one condition (long wall-clock) that makes them necessary. The panel flagged this as the single structural change required versus feature 005's design.

**Independent Test**: Start a faux-provider fusion that takes a known duration; verify a SQLite row exists for its job-state immediately after kickoff. Mid-fusion, restart the process; verify (a) a subsequent `fusion({ _resume_from })` returns a defined `{ status: "interrupted" | "not_found", instruction }` rather than crashing or hanging, (b) for a fusion that completed just before a restart, the result is still retrievable from SQLite within TTL, (c) live progress (candidate index) is absent after restart (no stale "running" rows). Verify the same durability holds for both parallel and sequential mode (no mode-specific storage branch).

**Acceptance Scenarios**:
1. **Given** a fusion has been kicked off, **When** the process is still running, **Then** a durable (SQLite) job-state record exists for its reference id from kickoff onward, identically for parallel and sequential mode.
2. **Given** a fusion completed and its result is within TTL, **When** the process restarts and the agent calls `fusion({ _resume_from })`, **Then** the completed result is returned from durable storage.
3. **Given** a fusion was in-flight when the process restarted, **When** the agent calls `fusion({ _resume_from })` after restart, **Then** the call returns a defined outcome (e.g. `{ status: "interrupted" | "not_found", instruction: <re-run or check dashboard> }`) — never an unhandled error or hang.
4. **Given** any fusion (parallel or sequential), **When** its live progress is queried after a restart, **Then** no stale "running" progress is reported (live progress is ephemeral and does not survive restart); only durable job-state/result does.

### Edge Cases

- **`_resume_from` with an unknown or expired reference id**: the call returns a defined `{ status: "not_found" | "expired", instruction: <re-run with original query> }`. Never an unhandled error. The agent's behavior on an evicted ticket is defined, not hallucinated.
- **Agent gives up (never retrieves a completed result)**: the job still ran and burned provider tokens. An explicit expiry error shape and a logged "never-retrieved" counter make this observable; the result is eventually evicted at TTL. This is the inherent cost of LLM-mediated retrieval and is documented, not eliminated.
- **Concurrent fusions from the same client**: each kickoff returns its own reference id; retrievals are independent. Spec 007 already permits concurrent fusions; this feature reuses the same per-job identity and adds no global ordering.
- **Concurrent `fusion({ prompt })` and `fusion({ _resume_from })` arguments**: when `_resume_from` is present, the original `prompt`/`context`/`persona` args are optional and ignored if supplied (the original inputs are bound to the reference id at kickoff). The tool schema must not force the agent to resend the full prompt on every poll.
- **TTL eviction vs. late completion race**: a job whose TTL expires near the moment of completion must not orphan-write or discard the result. The design includes a write-late guard (check the registry for eviction before storing) or a TTL extension while status shows running — finalized in planning.
- **Stalled candidate (parallel) with no progress for a long time**: the next `_resume_from` must eventually return an error outcome rather than empty long-polls forever — a stalled-job circuit (if no progress for N seconds, error out). This bounds the worst-case retrieval loop.
- **Judge failure after candidates complete**: the durable job-state must distinguish "candidates done, judge failed" from "all candidates failed," so a retrieval returns a meaningful error (and, where viable, access to raw candidate outputs rather than a whole-fusion failure) rather than a generic timeout.
- **Live progress after restart**: deliberately absent — persisted progress would be stale. Only durable job-state/result survive restart.
- **Tasks-aware client coexistence**: a Tasks client still gets the `CreateTaskResult` path; a non-Tasks client gets `_resume_from`. Both share the same detached runner and durable store. If a future client supports Tasks, the `_resume_from` branch becomes unused but is not removed (graceful coexistence).
- **Sequential mode not yet implemented (spec 007)**: this feature's parallel-mode path (US1) and durability (US3) are fully valuable on their own and ship first. The sequential-mode retrieval shape (US2) is implemented once spec 007's serial budget is available; until then, fusions are parallel-only and US2 is a no-op. This sequencing is noted in Assumptions, not a gap.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When a fusion is requested through a non-Tasks client (no `params.task` on `tools/call`), the system MUST return immediately from the kickoff call with a reference id and an imperative instruction to retrieve the result later via the same `fusion` tool — it MUST NOT hold the kickoff call open for the fusion's wall-clock.
- **FR-002**: The system MUST support a `_resume_from` parameter on the `fusion` tool. When present, the call MUST retrieve the state/result of the referenced fusion rather than start a new one, and the original `prompt`/`context`/`persona` arguments MUST be optional and ignored.
- **FR-003**: A `_resume_from` retrieval MUST return one of a small, fixed set of outcomes: `processing` (still running), `completed` (with the synthesized answer), or a defined failure outcome (`interrupted`, `not_found`, `expired`, or a fusion-level error) — never an unhandled exception or hang.
- **FR-004**: In **parallel mode**, a `_resume_from` retrieval that finds the fusion still running MUST bounded-long-poll up to a configured sub-ceiling wait (default ~40 s), returning `completed` if the fusion finishes within the wait, else `processing`. The wait MUST stay safely under the known client per-call ceiling.
- **FR-005**: The system MUST derive the kickoff message shape, retrieval cadence, and pending-wording from the execution mode. Parallel mode MUST use tight bounded long-polling with terse transparent-pacing wording (a retrieval mandate + an explicit `retry_after_ms`, no "do not inform the user" directive — that risks tripping safety-training refusals in frontier models); sequential mode MUST use ETA-guided (not tight) retrieval with user-facing "this is a long job, here is the dashboard link" wording. Both modes MUST surface a `retry_after_ms` pacing signal so the agent doesn't tight-loop or guess.
- **FR-006**: The reference id MUST be the same identity as the durable fusion/activity record — there MUST be exactly one id space, not a separate task-id ↔ activity-id ↔ reference-id mapping.
- **FR-007**: Job-state and, on completion, the synthesized result MUST be persisted to durable (SQLite) storage from kickoff onward, identically for parallel and sequential mode. This revises feature 005's non-durability stance for the retrieval use case.
- **FR-008**: The system MUST apply a time-to-live (TTL) to completed results. A retrieval after TTL MUST return `{ status: "expired", instruction: <re-run> }`. The TTL MUST be derived to comfortably exceed the longest legitimate fusion wall-clock (including sequential mode) plus margin — it MUST NOT be a fixed short value calibrated only for parallel fusions.
- **FR-009**: After a process restart, a `_resume_from` retrieval for an in-flight (interrupted) fusion MUST return a defined `interrupted`/`not_found` outcome with a re-run instruction; a retrieval for a fusion that completed before the restart MUST return its result from durable storage as long as it is within TTL.
- **FR-010**: Live candidate-by-candidate progress (e.g. "candidate 3 of 5") MUST remain ephemeral (in-memory) for both modes and MUST NOT be persisted — on restart, no stale running-progress is reported. (This is the only deliberately non-durable surface.)
- **FR-011**: The system MUST guard the TTL-eviction-vs-late-completion race (e.g. check eviction state before storing a late result, or extend TTL while status is running) so a result landing near expiry is never orphan-written or silently discarded.
- **FR-012**: The system MUST include a stalled-job circuit: a parallel fusion that shows no progress for a configured duration MUST cause the next `_resume_from` to return an error outcome rather than continue empty long-polling indefinitely.
- **FR-013**: The Tasks-aware client path (feature 005) MUST continue to work unchanged — it returns `CreateTaskResult` and is retrieved via `tasks/result`. The Tasks path and the `_resume_from` path MUST share the same detached runner and the same durable result store.
- **FR-014**: On judge failure after candidates complete, the durable job-state MUST distinguish a judge-side failure from an all-candidates-failed outcome, and the retrieval MUST return a meaningful error (and, where viable, access to raw candidate outputs) rather than a generic timeout.
- **FR-015**: The Tasks path (feature 005) and this `_resume_from` path MUST NOT change fusion semantics — candidate fan-out, the ≥2-survivor gate, and the two-step judge behave identically regardless of which retrieval path a client uses. Only *when* the kickoff call returns differs.

### Key Entities *(include if feature involves data)*

- **Deferred Fusion Job (durable record)**: The persistent state of a detached fusion, keyed by a single id that is also the activity id. Holds status (`processing` | `completed` | `interrupted` | `expired` | `error`), the synthesized result on completion, the original inputs (prompt/context/persona), timestamps (created, completed, expires), and execution mode. Survives restart. Replaces feature 005's in-memory task map for the retrieval use case. Live candidate-progress is deliberately not part of this record.
- **Reference Id (identity)**: The single id space for a fusion across kickoff, retrieval, logging, and the dashboard. Equal to the SQLite activity id; not a separate task id. Surfaced to the agent in the kickoff result and accepted as `_resume_from`.
- **Retrieval Shape (mode-dependent protocol)**: The set of fields and wording returned by a kickoff or retrieval, parameterized by execution mode. Parallel shape: tight bounded long-poll, terse transparent-pacing wording + `retry_after_ms`. Sequential shape: ETA-guided, user-facing long-job wording with a dashboard link + `retry_after_ms`. Both carry a reference id, a `retry_after_ms` pacing signal, and a retrieval instruction.
- **Ephemeral Live Progress (runtime-only state)**: In-memory candidate-level progress for an in-flight fusion (current candidate index, remaining count), consumed by the dashboard's live-status surface (spec 007) and not by retrieval. Not persisted; dies with the process.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A fusion whose wall-clock exceeds the non-Tasks client's per-call ceiling completes and returns its synthesized answer to the user with no client-side timeout — measured by a real end-to-end call through a non-Tasks client (re-scoping spec 005's never-run E1) succeeding where it previously timed out.
- **SC-002**: For a ~90 s parallel fusion on a non-Tasks client, the total number of LLM round-trips (kickoff + retrievals) stays in the low single digits — measured by the retrieval-call count for a representative job remaining at or below ~3.
- **SC-003**: A `_resume_from` retrieval never blocks for the fusion's full wall-clock — measured by every retrieval call returning within the configured bounded long-poll wait (~40 s) regardless of the fusion's total duration.
- **SC-004**: A process restart mid-fusion produces a defined, recoverable outcome — measured by a post-restart retrieval returning a documented `interrupted`/`not_found`/`completed` status (never a hang or unhandled error), and a pre-restart-completed result remaining retrievable within TTL.
- **SC-005**: For a sequential fusion (spec 007), the retrieval cadence is ETA-guided and the retrieval-call count stays in the single digits for a ~15-minute job — measured by the call count not scaling with wall-clock (unlike tight polling) and the kickoff surfacing an ETA and dashboard link.
- **SC-006**: Fusion semantics are identical regardless of retrieval path — measured by the synthesized answer for given inputs being byte-identical whether the fusion is fetched via `_resume_from`, via the Tasks path, or (hypothetically) synchronously; and by the activity log being indistinguishable across paths.
- **SC-007**: The Tasks-aware path (feature 005) continues to work unchanged — measured by a Tasks client still receiving a `CreateTaskResult` and retrieving via `tasks/result`, with no regression introduced by this feature.

## Assumptions

- **Non-Tasks clients are the primary audience**: codex and ZCode hardcode `task: None` and handle only `CallToolResult` (verified in codex source). This feature exists because feature 005 cannot help them. Tasks-aware clients keep working via feature 005's path; they are supported but not the target.
- **Client timeout is per-call and resets between calls**: the bounded long-poll escape depends on each `tools/call` getting a fresh ceiling, not a session/turn-level budget across all calls. This is the one foundational assumption that the panel flagged as unverified; it is confirmed empirically in planning (trace the client source for any wrapping turn/session deadline) before retrieval timing is locked.
- **Parallel mode ships first; sequential retrieval is gated on spec 007**: US1 (parallel retrieval) and US3 (durability) are valuable on their own and implementable today. US2 (sequential retrieval shape) requires spec 007's serial budget and live-status surface; until spec 007 lands, fusions are parallel-only and US2 is a documented no-op rather than a gap.
- **Storage is durable for both modes, progress ephemeral for both modes**: revises feature 005's non-durability acceptance. The cost (one extra SQLite row per fusion plus a result payload) is negligible for a local single-user tool (Constitution VII); the robustness gain (recoverable long jobs) is essential. The only deliberate non-durability is live candidate-progress, which on restart describes nothing.
- **LLM-mediated polling is an accepted trade-off for non-Tasks clients**: this is the pattern SEP-1686 rejected, but only because a protocol alternative exists. For non-Tasks clients it does not, so the cost (token spend + per-poll latency + reliance on the agent following the instruction) is accepted in exchange for a retrieval that actually works. The cost is bounded by the mode-aware cadence (tight for parallel, ETA-guided for sequential).
- **The ~60 s client ceiling is a conservative constant**: the default sub-ceiling long-poll wait (~40 s) and the kickoff's immediate return are sized with margin against the widely-reported ~60 s codex default. The exact wait is finalized in planning after the per-call-vs-session assumption is verified.
- **Feature 005's detached runner is reused, not replaced**: `startDetachedFusion` already runs the work out-of-band. This feature adds the durable store, the `_resume_from` retrieval, the mode-aware shapes, and the durability/restart handling on top of that runner. The Tasks egress and the `_resume_from` egress are two branches over one backend.
- **Scope excludes latency reduction and shell-based retrieval**: making a fusion fit under the ceiling by cutting candidates or judge steps is a quality regression and is explicitly out of scope (deferred to a possible future "quick mode"). Requiring the agent to shell out to `curl` a REST endpoint is rejected (crossing the tool→shell boundary is fragile and assumes shell access), though the kickoff result may include a dashboard URL as a human fallback.
- **No multi-reader semantics in v1**: a reference id is retrieved by the agent that kicked it off (or by the user via the dashboard). Sub-agent fan-out on one ticket, copied tickets, and read-many contracts are out of scope for v1 (YAGNI — Constitution VII).
- **No zombie-job cancellation in v1 (M7)**: if a client crashes after kickoff but before retrieval, the detached fusion runs to completion (burning provider tokens for up to ~21 min in sequential mode) with no user benefiting. True cancellation requires an `AbortController` wired through `runFusion`, which is explicitly out of scope (the plan adds no `AbortController` — consistent with 005/007). Mitigations in v1: (a) the stalled-job circuit (FR-012) catches *hung* jobs; (b) the never-retrieved counter (F3) makes the cost observable; (c) the cost is bounded — one-time provider spend per abandoned job, capped by TTL. Active abandonment-driven cancellation is scoped to a possible v1.1 addition alongside the broader `AbortController` work.
